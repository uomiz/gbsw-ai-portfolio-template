## 📚 RAG 완전 이해하기

### 일반 LLM vs RAG

#### ❌ 일반 LLM (문제점)

```
사용자: "내가 어떤 프로젝트를 했는지 알려줘"
      ↓
    LLM (학습 데이터에만 의존)
      ↓
답변: "죄송하지만 당신의 프로젝트 정보를 알 수 없습니다."
또는: "아마도 이런 프로젝트를 했을 것 같습니다." (환각!)
```

**문제:**
- LLM은 당신의 프로젝트를 모름
- 추측해서 거짓 정보 생성 (Hallucination)

#### ✅ RAG (해결책)

```
사용자: "내가 어떤 프로젝트를 했는지 알려줘"
      ↓
1. 벡터 검색 → projects.json에서 관련 문서 찾기
      ↓
2. 검색된 근거를 LLM에게 제공
      ↓
3. LLM이 근거 기반으로 답변 생성
      ↓
답변: "당신은 2개의 프로젝트를 수행했습니다:
       1) Interlocutor Adaptation... (출처: projects.json)"
```

**장점:**
- 사실 기반 답변
- 출처 제공으로 신뢰도 ↑
- 최신 정보 반영 가능

## 🔬 RAG 작동 원리 상세 분석

### Step 1: 임베딩 (Embedding)

**임베딩이란?**
텍스트를 숫자 벡터(배열)로 변환하는 과정

```python
# 예시: 문장을 벡터로 변환
"NLP 프로젝트" → [0.2, 0.8, 0.1, ..., 0.4]  # 384차원 벡터
"자연어처리 연구" → [0.21, 0.79, 0.12, ..., 0.39]  # 비슷한 벡터!

"음식 사진" → [0.9, 0.1, 0.2, ..., 0.7]  # 완전 다른 벡터
```

**핵심:**
- 의미가 비슷한 문장 = 비슷한 벡터
- 벡터 간 거리로 유사도 측정 가능

**우리 시스템:**
- 모델: `sentence-transformers/all-MiniLM-L6-v2`
- 차원: 384차원 벡터
- 언어: 영어/한국어 모두 지원

### Step 2: 벡터 DB 구축 (최초 1회)

```python
# 1. projects.json 로드
projects = [
    {"title": "프로젝트 A", "description": "NLP 관련..."},
    {"title": "프로젝트 B", "description": "CV 관련..."}
]

# 2. 각 프로젝트를 임베딩
프로젝트 A description → 벡터 A [0.2, 0.8, ...]
프로젝트 B description → 벡터 B [0.9, 0.1, ...]

# 3. ChromaDB에 저장
chroma_db/
  ├── 벡터 A (+ 메타데이터: 프로젝트 A)
  └── 벡터 B (+ 메타데이터: 프로젝트 B)
```

### Step 3: 질문 검색 (매번 실행)

```python
# 사용자 질문
질문: "NLP 프로젝트는 뭐가 있어?"

# 1. 질문도 임베딩
질문 벡터 = [0.21, 0.79, 0.11, ..., 0.41]

# 2. ChromaDB에서 유사도 계산 (코사인 유사도)
질문 벡터 vs 벡터 A: 유사도 0.92 ← 높음!
질문 벡터 vs 벡터 B: 유사도 0.15 ← 낮음

# 3. top_k=2라면 상위 2개 반환
검색 결과: [프로젝트 A (0.92), 프로젝트 B (0.15)]
```

### Step 4: 프롬프트 구성

```python
# 검색된 문서를 프롬프트에 삽입
프롬프트 = f"""
당신은 AI 포트폴리오 질의응답 전문가입니다.

[제공된 문서]
프로젝트 A: NLP 관련 대화 시스템을 개발했습니다...
프로젝트 B: Computer Vision 객체 인식을 연구했습니다...

[질문]
NLP 프로젝트는 뭐가 있어?

[지침]
문서 내용만을 근거로 답변하세요.
"""
```

### Step 5: LLM 답변 생성


## 🧪 실습 1: top_k 파라미터 실험

### top_k란?

**정의:** 검색할 문서의 개수

```python
top_k=1  # 가장 유사한 1개만
top_k=2  # 상위 2개
top_k=5  # 상위 5개
```

### 실험 설계

[server/rag_app.py:63](../server/rag_app.py#L63) 수정:

```python
rag_engine = build_rag_engine(
    projects_json_path=str(projects_json),
    gemini_api_key=gemini_api_key,
    rebuild=False,
    top_k=1,  # ← 여기를 변경!
    chunk_size=1000,
    chunk_overlap=0
)
```

### 실험 절차

#### 실험 A: top_k=1

1. **설정:**
   ```python
   top_k=1
   ```

   ```bash

   ```

3. **테스트 질문:**
   ```
   질문: "어떤 프로젝트를 했나요?"
   ```

4. **결과 기록:**
   ```
   답변: ...
   출처: [1개만 표시됨]

   관찰:
   - 정보가 부족한가?
   - 한 프로젝트만 언급하는가?
   ```

#### 실험 B: top_k=3

1. **설정 변경:**
   ```python
   top_k=3
   ```

2. **재실행:**


### 실험 결과 분석

#### 성능 비교표 작성

| 설정 | 검색 품질 | 답변 완성도 | 응답 속도 | 종합 평가 |
|------|-----------|-------------|-----------|-----------|
| top_k=1 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 정보 부족 |
| top_k=2 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | **최적** |
| top_k=3 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | 약간 과다 |
| top_k=5 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | 노이즈 많음 |

### 결론 도출 가이드

**top_k 선택 기준:**

```
프로젝트 개수가 2~3개 → top_k=2
프로젝트 개수가 5~10개 → top_k=3
프로젝트 개수가 10개 이상 → top_k=3~5

원칙: 전체의 20~40% 정도가 적당
```

## 📏 실습 3: 청크 크기 실험

### 청킹(Chunking)이란?

긴 문서를 작은 조각으로 나누는 것

```python
# 원본 문서 (2000자)
프로젝트 설명: "이 프로젝트는 NLP 기반 대화 시스템입니다... (2000자)"

# chunk_size=1000으로 분할
Chunk 1: "이 프로젝트는 NLP 기반 대화... (1000자)"
Chunk 2: "...시스템의 특징은 다음과 같습니다... (1000자)"
```

### 왜 청킹이 필요한가?

1. **LLM 입력 제한:**
   - Gemini Flash 입력: 최대 32K 토큰
   - 문서가 너무 크면 일부만 전달

2. **검색 정확도:**
   - 작은 청크 → 더 정확한 검색
   - 큰 청크 → 관련 없는 내용 포함

### chunk_size vs chunk_overlap

```python
# 예시: chunk_size=50, chunk_overlap=10

원본: "ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"

Chunk 1: "ABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789ABCDEFGHIJKL"  (50자)
                                              ↓ 겹침 10자
Chunk 2:                          "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ..."
```

**overlap을 쓰는 이유:**
- 문맥이 끊기는 것을 방지
- 청크 경계에 걸친 정보 보존

### 실험 설계

[server/rag_app.py:66](../server/rag_app.py#L66) 수정:

```python
rag_engine = build_rag_engine(
    projects_json_path=str(projects_json),
    gemini_api_key=gemini_api_key,
    rebuild=True,  # ⚠️ 반드시 True!
    top_k=2,
    chunk_size=500,  # ← 여기 변경!
    chunk_overlap=0  # ← 여기도 실험!
)
```

### ⚠️ 중요: rebuild=True

**왜 필요한가?**
- chunk_size 변경 → 벡터 DB를 새로 만들어야 함
- rebuild=False면 기존 DB 사용 (변경 반영 안 됨)

**주의:**
- rebuild=True → 1~2분 소요 (다시 임베딩)
- 실험 후 다시 rebuild=False로 변경
